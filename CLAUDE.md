## Project Overview

Kubernetes deployment for [Pimcore](https://pimcore.com/) (PHP data/experience management platform) targeting Minikube (local dev) and Google Kubernetes Engine (GKE). Uses [Helmsman](https://github.com/Praqma/helmsman) (a Helm state manager) to orchestrate multiple Helm releases.

Stack: Nginx → PHP-FPM (Pimcore) → MariaDB + Redis

## Deployment Commands

```bash
# Generate .env file with secure random passwords (run once, never overwrites)
./scripts/generate-dotenv.py

# Deploy to Minikube
helmsman -apply -f helmsman/dsf/minikube.yaml

# Deploy to GKE
helmsman -apply -f helmsman/dsf/gke.yaml

# Destroy all releases
helmsman -destroy -f helmsman/dsf/[minikube|gke].yaml
```

## Maintenance Shell

`scripts/maintenance-shell.sh` is a wrapper that scales up the maintenance pod (0 replicas at rest), runs commands, then optionally scales back down.

```bash
# Interactive shell
./scripts/maintenance-shell.sh shell

# Run a maint-* command in the pod (auto-prefixed with "maint-")
./scripts/maintenance-shell.sh cache-reset
./scripts/maintenance-shell.sh db-import dump.sql.gz
./scripts/maintenance-shell.sh list-db-backups

# Download latest DB backup locally
./scripts/maintenance-shell.sh --download-latest-db-backup

# Run arbitrary command
./scripts/maintenance-shell.sh -- php bin/console pimcore:cache:clear

# Scale down the maintenance shell without running anything
./scripts/maintenance-shell.sh --scale-down
```

Environment variable overrides: `MAINT_SHELL_DEPLOYMENT`, `MAINT_SHELL_INSTANCE`, `MAINT_APP_NAME`.

## Architecture

### Helm Charts (`charts/`)

- **`charts/pimcore/`** — Main chart containing all application components:
  - `templates/php/` — PHP-FPM deployment
  - `templates/nginx/` — Nginx reverse proxy (includes `backendconfig.yaml` for GKE)
  - `templates/maintenance/` — Maintenance shell deployment (0 replicas at rest), background worker, Pimcore cronjob, MySQL backup cronjob, generic console command cronjobs
  - `templates/hooks/` — Helm Jobs that run on install/upgrade: `install.yaml` (composer create-project / clone), `initialize.yaml`, `migrate.yaml`
  - `templates/secret/` — Secret resources
- **`charts/mysql/`** — Bitnami MariaDB chart v16.4.0 (pinned by digest for stability)
- **`charts/redis/`** — Bitnami Redis chart v18.16.1 (pinned by digest for stability)

Key `values.yaml` knobs:
- `pimcore.appEnv` — defaults to `dev`; change to `prod` for production
- `pimcore.createProject` — `pimcore/skeleton` (default) or `pimcore/demo`; update `pimcore.customConfigFiles` paths if changed
- `pimcore.customConfigFiles` — mount `config.yaml` / `bundles.php` as ConfigMaps instead of editing the PV directly (toggle `enabled: true` in DSF)
- `pvc.data.initFromRepo` — instead of running `composer create-project`, clone an existing Git repo; set `gitRepositoryUrl` and optionally `gitPersonalAccessToken`
- `maintenance.consoleCronjobs` — map of scheduled `bin/console` command CronJobs; each key becomes a CronJob named `<release>-<key>` with `enabled`, `schedule`, and `command` fields (plus optional `resources`, `concurrencyPolicy`, `activeDeadlineSeconds`, etc.)
- `maintenance.mysqlBackup.compression.tool` — compression binary for DB backups (`pigz`, `gzip`, `xz`, etc.); defaults to `pigz`
- PHP image `pimcore/pimcore:php8.2-latest`, 4 CPU / 8Gi memory, PHP memory limit 512M, dynamic PM pool (3–8 spare, 100 max children)

### Helmsman DSF (`helmsman/dsf/`)

- `minikube.yaml` — Local dev; uses `minikube` kubeContext, config files via ConfigMaps
- `gke.yaml` — GKE; uses custom StorageClass `pimcore-gke-retained`, GKE BackendConfig, Google Cloud Secrets Manager integration

### Lifecycle Hooks (`helmsman/hooks/`, `custom-hooks/`)

`helmsman/hooks/hook-handler.sh` parses the hook invocation name to extract cloud provider and lifecycle event, applies cloud-specific K8s configs from `helmsman/cloud-configs/`, then runs user-defined scripts from `custom-hooks/<provider>-<event>/` using `run-parts` (lexicographical order).

To add custom hook logic, drop executable scripts into `custom-hooks/gke-pre-install/` (or equivalent). See `custom-hooks/README.md` and `examples/custom-hooks/` for examples.

### Environment Variables (`.env`)

Generated by `scripts/generate-dotenv.py`. Referenced in DSF files via `{{ env "VAR_NAME" }}`. Contains DB passwords, Redis password, and app secret. File permissions set to 0o600.

## Known Pitfalls

- **Stale containers after chart upgrades**: If the `mysql-backup-cronjob` pod shows an extra `maintenance` container alongside `mysql-backup`, it is a remnant from an older chart version. Both containers run `mysqldump` simultaneously, causing memory exhaustion and silent OOMKill on GKE Autopilot. Fix: re-apply the chart (`helmsman -apply`) to reconcile the CronJob spec.
- **Bitnami legacy images**: MariaDB and Redis use `bitnamilegacy/` images pinned by digest. If pulling new images, update the digest in the DSF — do not just change the tag.
